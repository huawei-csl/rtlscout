## Bugs found and fixed

### Bug 1: sBox_8 produces enc_dec-dependent garbage on the off-axis output

The 250-LOC GF-math `sBox_8` produces BOTH `sbox_out_enc` and
`sbox_out_dec` from a SHARED input pipeline whose isomorphism
transform depends on the `enc_dec` input:

```
base_new_enc, base_new_dec = isomorphism(sbox_in)       # comb
base_new = ~(enc_dec ? base_new_enc : base_new_dec)     # depends on enc_dec
out_gf_inv8_stage1 = gf_inv_8_stage1(base_new)
[pipeline reg captures both]
out_gf_inv8_1 = gf_inv_8_stage2(base_new_pp, out_gf_pp)
sbox_out_enc = ~isomorphism_inv(out_gf_inv8_1, ENC)     # always ENC iso
sbox_out_dec = ~isomorphism_inv(out_gf_inv8_1, DEC)     # always DEC iso
```

When `enc_dec=ENC=1`: `sbox_out_enc = AES_SBOX[sbox_in]` and
`sbox_out_dec` is "garbage" (input through ENC isomorphism, then DEC
inverse — doesn't simplify to a meaningful function).
When `enc_dec=DEC=0`: roles flip.

Downstream `key_expander.g_out = sbox_out_enc` ALWAYS. So when in DEC
mode, key_expander sees garbage; that garbage flows into round keys.

My first attempt used a plain LUT producing clean AES SBOX values
regardless of `enc_dec` — wrong; the failure pattern was identical
`0x08080808` XOR vs expected `key_bus`.

**Fix:** 4 LUTs indexed by (`sbox_in`, `enc_dec`):
- `SBOX_ENC_OUT_WHEN_ENC[i]` = AES_SBOX_ENC[i] (standard)
- `SBOX_ENC_OUT_WHEN_DEC[i]` = garbage
- `SBOX_DEC_OUT_WHEN_ENC[i]` = garbage
- `SBOX_DEC_OUT_WHEN_DEC[i]` = AES_SBOX_DEC[i] (standard)

Tables generated by Python-simulating the verilog GF math
byte-by-byte — see [`compute_sbox_tables.py`](compute_sbox_tables.py).
Sanity check: 256/256 diagonal entries match AES_SBOX.

Added a 1-bit `ed_pp_r` register (shared across all 4 byte lanes,
since verilog's sBox_8 instances all share the same `enc_dec` input)
that captures `enc_dec` along with the byte. Output then 4-way muxes
between the 4 tables.

Result: 953 → 1011 vector pass.

### Bug 2: `cat` LSB-first on `sbox_sel_mux`

Verilog: `sbox_sel_mux = sbox_sel | {1'b0, col_sel_host}` — the
`{1'b0, col_sel_host}` is a 3-bit zero-extension of the 2-bit
`col_sel_host`.

My first port: `sbox_sel_mux = sbox_sel | cat(Const(0, UInt(1)), col_sel_host)`.
Spirehdl `cat` is LSB-first: this puts `Const(0)` at the LSB, so it's
`col_sel_host << 1`, not a zero-extension.

Fix: `sbox_sel_mux = sbox_sel | cat(col_sel_host, Const(0, UInt(1)))`
(zero at MSB end).

Result: 1011 → 1569 vector pass. Huge improvement because this bug
caused `sbox_input` to pick the wrong source register (often a `col[i]`
that wasn't loaded, hence the prevalent `col_bus = 0` failures).

### Bug 3: `rc_dir` / `rc_inv` defaults wrong for round >= 10

Verilog `rc_dir`:
```verilog
if(round == 8) rc_dir = 8'h1b;
else if(round == 9) rc_dir = 8'h36;
else rc_dir = 8'h01 << round;
```

For `round` in `[0..7]`: `1 << round` = `0x01..0x80`.
For `round` in `[10..15]`: `1 << round` overflows 8 bits → `rc_dir = 0`.

My port:
```python
rc_dir = Const(1, UInt(8))     # default for "no match in mux chain"
for r in range(8):
    rc_dir = mux(round_4 == Const(r, UInt(4)), Const(1 << r, UInt(8)), rc_dir)
```

For `round` in `[10..15]`: my mux chain has no match → returns the
default `Const(1)`. But verilog gives 0.

**Fix:** change the default from `Const(1)` to `Const(0)` (which is
the correct value for round >= 10, where 1<<round overflows).

Same fix for `rc_inv` (default changed from `0x80` to `0`).

This bug was hard to spot because it only affects `round >= 10`, and
the failures looked like single-bit-toggled `key_bus` outputs. Found
by spirehdl-Simulator trace: at vec=3 with round=15, my g_func MSB
XORed with rc=1 instead of rc=0, toggling bit 0 in MSB byte of
downstream key registers.

Result: 1569 → 2000 vector pass. ✓

## How I found Bug 3 (the spirehdl Simulator trace)

The trace at vec=5 showed `key_3 = 0d080808` (mine) vs `0c080808`
(expected). Working backward:
- `key[3]` at vec=5 = `ko_3 = key[3]_pre ^ key[2]_pre` (ko_3 in the
  key_expander).
- pre-edge values: my `key[3]_pre=6a6b6b6b`, `key[2]_pre=67636363`.
- XOR = `0d080808` ✓ (my output).
- For expected: `key[2]` must have been `66636363` (off by 1 in MSB byte).
- That divergence traces back to vec=4 where `key[2]` was updated
  using a different g_func.
- g_func MSB byte = g_out_MSB ^ rc.
- At the relevant time, `round_pp1 = 15` and my rc_dir for round=15
  returned 1 (the default) instead of 0 (what verilog does because
  `8'h01 << 15` overflows to 0 in 8 bits).

The 1-bit XOR pattern in single bytes was the key signal — pointed at
"rc differs by 1 bit between mine and verilog".
